# PlanTick 实施计划

## 1. 目标

本计划基于当前已确定的技术路线：
- `Web + PWA`
- `React + TypeScript + Vite`
- `IndexedDB`
- `Supabase`

目标是在尽量低复杂度下，先做出一个单用户可长期使用的待办 + 日历应用，并把架构关键点前置验证，避免在工作区接入和同步模型上返工。

## 2. 已锁定决策

- 只做 `Web/PWA` 主线，不做原生端
- 只服务单一用户，不做多人协作
- 先做本地优先，再做多设备同步
- 工作区口令由 `Edge Function` 在服务端校验
- 多设备同步以“前台补拉”为主，`Realtime` 为增强项
- 冲突规则统一为最后更新时间覆盖
- 重复待办保留在 V1，但采用“完成当前实例后生成下一实例”
- 重复待办的下一实例由服务端统一派生，客户端不直接生成

## 3. 实施原则

- 所有设计同时兼顾桌面浏览器、手机浏览器和 PWA
- 先解决工作区接入与数据隔离，再做业务功能
- 先固化数据模型、同步契约和本地存储结构，再做页面交互
- UI 读取以本地投影结果为主，不把核心页面渲染绑定到网络请求
- 每个阶段都要求存在明确交付物和退出条件

## 4. 核心实现方案

### 4.1 工作区接入

- 不提供注册/登录页面
- 用户通过“创建工作区”或“输入工作区口令进入工作区”接入同一份数据
- 前端仅持有 Supabase 匿名会话
- `Edge Function` 负责：
  - 创建工作区
  - 校验口令
  - 将当前设备加入 `workspace_members`
- 后续所有远端读写都以 `workspace_id` 为边界，并通过 `workspace_members` 做隔离

### 4.2 本地数据与同步

- 业务写入顺序固定为：
  - 先写 `IndexedDB`
  - 再写本地 `outbox`
  - 联网后批量推送到 `Supabase`
- 本地数据库至少包含：
  - `todos`
  - `categories`
  - `events`
  - `outbox`
  - `sync_meta`
  - `workspace_meta`
- 远端同步以增量补拉为主，触发时机包括：
  - 进入应用
  - 页面回到前台
  - 网络恢复
  - 推送成功后
- `updated_at` 以服务端时间为准
- `deleted` 保留软删除语义，不做物理删除

### 4.3 重复待办

- V1 仅支持：
  - 按天重复
  - 按周重复
  - 按月重复
- 客户端完成重复待办时，只提交“完成当前实例”操作
- 服务端原子地：
  - 标记当前实例完成
  - 生成下一实例
  - 返回最终结果
- 这样可以避免双设备同时完成同一待办时生成多条下一实例

## 5. 分阶段实施

### Phase 0：Web/PWA 与工作区接入可行性验证

目标：
- 验证 Web 技术路线、PWA 基线和工作区接入方案均可落地

任务：
- 初始化 `Vite + React + TypeScript` 项目
- 接入 `vite-plugin-pwa`
- 验证 Windows 浏览器访问
- 验证手机浏览器访问
- 验证 PWA 安装能力
- 验证 `IndexedDB` 可读写
- 建立最小 `Supabase` 表结构
- 验证匿名会话可调用 `Edge Function`
- 打通“创建工作区 / 输入口令加入工作区 / 读取测试数据”最小闭环

交付物：
- 可运行的网页应用骨架
- 可安装的 PWA 基线
- 已验证的工作区接入 Spike
- 已验证的 `IndexedDB` 和 `Supabase` 连通性

退出条件：
- Windows 与手机浏览器都能打开应用
- PWA 可安装
- 本地存储可用
- 可以创建工作区并使用口令在另一设备进入同一工作区

### Phase 1：数据模型与前端工程骨架

目标：
- 固化后续所有业务功能依赖的数据模型、同步契约和前端结构

任务：
- 设计 `workspaces`、`workspace_members`、`todos`、`categories`、`events` 远端结构
- 设计本地 `IndexedDB` schema
- 设计 `outbox` 事件格式
- 固化 `updated_at`、`deleted`、`id` 的约定
- 建立页面结构与路由
- 建立应用壳与通用布局
- 建立响应式规则
- 建立本地数据访问层
- 建立同步入口层

交付物：
- 稳定的前端工程骨架
- 本地数据层
- 远端访问层
- 明确的同步契约文档

退出条件：
- 本地表结构和远端表结构已固定
- `enqueue`、`push`、`pull`、`reconcile` 的职责已明确
- 项目结构可支撑后续模块独立开发

### Phase 2：工作区进入与本地优先基础能力

目标：
- 完成工作区进入流程和本地优先的应用启动闭环

任务：
- 实现首次创建工作区流程
- 实现输入口令进入工作区流程
- 建立本地工作区状态
- 建立本地缓存表
- 建立本地 `outbox`
- 建立同步状态记录
- 建立启动恢复流程
- 支持离线时恢复最近使用的工作区

交付物：
- 工作区进入流程
- 启动恢复流程
- 本地缓存与同步状态机制

退出条件：
- 单设备上可以进入工作区并使用本地数据
- 刷新页面后仍能恢复最近工作区和本地数据
- 离线情况下仍可完成本地读写

### Phase 3：待办与分类

目标：
- 完成待办管理主流程

任务：
- 分类创建、编辑、删除
- 待办列表
- 新建待办
- 编辑待办
- 完成/恢复待办
- 删除待办
- 截止日期设置
- 备注
- 分类筛选
- 重复规则

交付物：
- 可独立使用的待办模块
- 可独立使用的分类模块

退出条件：
- 单设备上待办与分类功能完整可用
- 重复待办在本地流程可闭环

### Phase 4：月视图与日程

目标：
- 完成日历主流程

任务：
- 月视图
- 截止待办映射到日历
- 手动日程创建
- 日程编辑与删除
- 桌面端和移动端日历布局适配
- 单日内容展开与查看

交付物：
- 可用的月视图日历
- 日程管理能力

退出条件：
- 用户可以通过月视图查看待办和日程
- 桌面端与移动端都能正常完成日程操作

### Phase 5：同步闭环与多设备

目标：
- 完成多设备同步闭环

任务：
- 实现本地 `outbox` 批量推送
- 实现远端增量补拉
- 实现本地与远端合并逻辑
- 实现软删除同步
- 实现重复待办的服务端派生
- 补充前台触发补拉的调度时机
- 视体验情况决定是否增加 `Realtime`

交付物：
- 多设备同步能力
- 冲突处理能力
- 重复待办跨设备一致性能力

退出条件：
- 两台设备之间数据可同步
- 离线后恢复联网可自动补齐
- 并发编辑按最后更新时间覆盖
- 重复待办不会因为双设备同时操作而生成多条下一实例

### Phase 6：加固与交付

目标：
- 将系统提升到可日常使用的稳定程度

任务：
- 弱网和失败重试处理
- `outbox` 重试与退避策略
- PWA 体验优化
- 页面刷新和恢复验证
- 移动端适配优化
- 基础测试与验收
- 部署与文档补全

交付物：
- 可日常使用的 V1 版本
- 基础验收文档

退出条件：
- 桌面与手机都可稳定使用
- 核心流程具备可重复验证的验收清单

## 6. 需要固定的接口与数据

### 6.1 Edge Function

- `workspace-create(passphrase)`
- `workspace-join(passphrase)`
- `sync-push(workspaceId, operations[])`
- `sync-pull(workspaceId, since)`
- `todo-complete-recurring(todoId)` 或等价的原子处理逻辑

### 6.2 本地类型

- `WorkspaceMeta`
- `SyncMeta`
- `OutboxItem`
- `Todo`
- `Category`
- `Event`

### 6.3 同步字段

- 每条业务数据至少包含：
  - `id`
  - `workspace_id`
  - `updated_at`
  - `deleted`
- `OutboxItem` 至少包含：
  - `id`
  - `entity_type`
  - `entity_id`
  - `operation`
  - `payload`
  - `created_at`
  - `retry_count`

## 7. 测试与验收

单元测试：
- 日期映射与月历投影
- 待办、分类、日程本地仓储 CRUD
- `outbox` 入队、去重、重试
- 最后更新时间覆盖合并逻辑
- 重复待办下一实例计算逻辑

集成测试：
- `IndexedDB` 初始化与迁移
- `Edge Function` 创建/加入工作区
- `sync-push` 与 `sync-pull`
- 服务端完成重复待办时只生成一个新实例

端到端测试：
- 新设备创建工作区并进入主应用
- 第二台设备输入同一口令进入同一工作区
- 设备 A 创建待办后，设备 B 补拉可见
- 设备 A 离线编辑后，恢复联网可同步到设备 B
- 两设备并发编辑同一待办，最终按服务端 `updated_at` 覆盖
- 月视图可同时展示待办截止项与手动日程

## 8. 推荐顺序总结

推荐执行顺序：
1. Web/PWA 与工作区接入可行性验证
2. 数据模型与前端工程骨架
3. 工作区进入与本地优先能力
4. 待办与分类
5. 月视图与日程
6. 同步闭环与多设备
7. 加固与交付
