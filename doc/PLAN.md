# PlanTick 实施计划

## 1. 目标

本计划基于当前已确定的技术路线：
- `Web + PWA`
- `React + TypeScript + Vite`
- `IndexedDB`
- `Supabase`

目标是在尽量低复杂度下，先做出一个单用户可长期使用的待办 + 日历应用。

## 2. 实施原则

- 先做网页主线，不做原生端
- 先做本地可用，再做同步
- 先做待办和月视图，再补体验细节
- 所有方案以桌面浏览器和手机浏览器/PWA 为准

## 3. Phase 0：Web/PWA 环境验证

目标：
- 验证 Web 技术路线与基础环境可用

任务：
- 初始化 `Vite + React + TypeScript` 项目
- 接入 `vite-plugin-pwa`
- 验证 Windows 浏览器访问
- 验证手机浏览器访问
- 验证 PWA 安装能力
- 验证 IndexedDB 可读写
- 验证与 Supabase 的最小连通性

交付物：
- 可运行的网页应用骨架
- 可安装的 PWA 基线
- 已验证的 IndexedDB 和 Supabase 连通性

退出条件：
- Windows 与手机浏览器都能打开应用
- PWA 可安装
- 本地存储可用
- Supabase 可连通

## 4. Phase 1：前端基础架构

目标：
- 建立可持续开发的前端结构

任务：
- 建立页面结构与路由
- 建立通用布局
- 建立状态管理
- 建立 UI 主题和响应式规则
- 建立 IndexedDB 数据访问层
- 建立 Supabase 接入层

交付物：
- 稳定的前端工程骨架
- 本地数据层
- 云端接入层

退出条件：
- 项目可稳定开发和扩展

## 5. Phase 2：工作区与本地数据

目标：
- 完成工作区进入流程和本地优先基础能力

任务：
- 实现工作区口令输入流程
- 建立本地工作区状态
- 建立本地缓存表
- 建立本地 outbox
- 建立同步状态记录

交付物：
- 工作区进入流程
- 本地缓存与同步状态机制

退出条件：
- 单设备上可以进入工作区并使用本地数据

## 6. Phase 3：待办功能

目标：
- 完成待办管理主流程

任务：
- 待办列表
- 新建待办
- 编辑待办
- 完成/恢复待办
- 删除待办
- 分类管理
- 截止日期设置
- 重复规则

交付物：
- 可独立使用的待办模块

退出条件：
- 单设备上待办功能完整可用

## 7. Phase 4：日历功能

目标：
- 完成日历主流程

任务：
- 月视图
- 截止待办映射到日历
- 手动日程创建
- 日程编辑与删除
- 桌面端和移动端日历布局适配

交付物：
- 可用的月视图日历
- 日程管理能力

退出条件：
- 用户可以通过月视图查看和管理数据

## 8. Phase 5：同步与多设备

目标：
- 完成多设备同步

任务：
- 设计 Supabase 表结构
- 建立工作区隔离
- 建立本地 outbox 推送逻辑
- 建立远端变更拉取逻辑
- 接入实时订阅或补拉逻辑
- 实现最后更新时间覆盖

交付物：
- 多设备同步能力
- 冲突处理能力

退出条件：
- 两台设备之间数据可同步
- 离线后恢复联网可自动补齐

## 9. Phase 6：加固与交付

目标：
- 将系统提升到可日常使用的稳定程度

任务：
- 弱网和失败重试处理
- PWA 体验优化
- 页面刷新和恢复验证
- 移动端适配优化
- 基础测试与验收
- 文档补全

交付物：
- 可日常使用的 V1 版本

退出条件：
- 桌面与手机都可稳定使用

## 10. 推荐顺序总结

推荐执行顺序：
1. Web/PWA 环境验证
2. 前端工程骨架
3. 工作区与本地数据
4. 待办模块
5. 日历模块
6. 同步模块
7. 加固与交付
