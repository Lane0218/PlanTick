# AGENTS

本文件定义本仓库中 Agent 的默认协作规则。除非用户在当前对话中明确覆盖，否则以下规则始终成立。

## 1. 对话语言

- 与用户交流时，必须始终使用中文。
- 包括需求确认、进度汇报、问题说明、结果总结，全部使用中文。

## 2. 运行环境

- 当前默认工作环境为 `WSL`。
- 在执行命令、处理路径、安装依赖、运行脚本时，必须优先按 `WSL/Linux` 环境理解。
- 涉及浏览器、桌面程序、移动端联调时，要明确区分：
  - `WSL` 内执行
  - Windows 主机执行
  - 手机端执行

## 3. 提交与推送要求

- 每次完成一次实际修改后，必须主动执行：
  - `git add`
  - `git commit`
  - `git push`
- 不要只修改代码而不提交。
- 不要把“是否需要提交和推送”留给用户再次提醒。

### 3.1 Commit Message 规范

- 提交信息必须使用明确、简洁、可检索的规范格式：

```text
type(scope): summary
```

- `type` 允许使用：
  - `feat`
  - `fix`
  - `docs`
  - `refactor`
  - `test`
  - `chore`
- `scope` 应尽量反映改动范围，例如：
  - `web`
  - `pwa`
  - `sync`
  - `docs`
  - `db`
  - `ui`
- `summary` 使用英文短句，简洁描述本次改动，不要写空泛内容。

示例：

```text
feat(web): initialize vite app shell
fix(sync): handle stale outbox replay
docs(agent): add repository collaboration rules
test(pwa): cover workspace bootstrap flow
```

## 4. 测试要求

- 每次完成修改后，必须主动判断并执行“合理且分层”的测试。
- 测试策略必须与改动范围匹配，不能一律跳过，也不能一律全量跑。

### 4.1 文档改动

- 如果只修改文档，例如：
  - `README`
  - `SPEC`
  - `PLAN`
  - `AGENTS`
- 则不需要运行测试。

### 4.2 小范围改动

- 如果改动范围较小，只影响局部模块：
  - 只运行对应模块相关测试
  - 只运行必要的构建检查
- 不要默认执行全量测试。

### 4.3 大范围改动

- 如果改动覆盖面较大，例如：
  - 涉及核心状态流转
  - 涉及同步逻辑
  - 涉及多个页面或多个模块
  - 涉及公共基础设施
- 则需要执行更完整的测试集。

### 4.4 端到端测试

- 允许并推荐使用 `Playwright` 进行端到端测试。
- 涉及关键用户流程时，应优先补充或执行对应的端到端测试。
- 如果当前环境不具备运行测试的条件，必须明确说明原因、影响范围和未验证项。

## 5. 实施原则

- 以当前仓库现行方案为唯一实现基线。
- 不在沟通和文档中主动回顾、比较或引入历史方案。
- 除非用户明确要求，否则不要扩展到当前目标之外的技术讨论。

## 6. 执行完成后的最小交付要求

- 完成修改
- 运行与改动范围匹配的测试
- 提交代码
- 推送到远程仓库
- 用中文向用户说明：
  - 做了什么
  - 运行了哪些测试，或为何不需要测试
  - 提交信息是什么
